# harness-design 사용 가이드

> Claude Code 하네스(CLAUDE.md / Permissions / Hooks / MCP / Subagent / 진화 원칙)를 단계적으로 진단·보강·진화시키는 스킬의 실전 사용법.

---

## 1. 언제 효과적인가 — 솔직한 가이드

### ✅ 가장 효과적 — 기존 프로젝트 검토·진화

이 스킬의 80% 가치는 여기서 나옵니다.

- **2주~수개월 써본 프로젝트** — 어떤 실수가 반복되는지, 어떤 규칙이 안 지켜지는지 데이터가 쌓인 상태
- **모델 업그레이드 직후** — 옛 모델용 규칙이 잔존하는지 재검토 필요
- **마일스톤·릴리스 종료 후** — "v1 출시 전까지만" 식 임시 규칙 정리
- **새 팀원 합류 시** — 암묵 규칙 명문화 / 팀 표준 재확인
- **클로드가 자꾸 같은 실수를 반복할 때** — 어디에 어떤 강제력으로 잡을지 결정

이런 경우 Phase 0 진단(✅/⚠️/❌ 표)이 가장 큰 가치를 만듭니다. 평가 결과 — 베이스라인 대비 통과율 +29.2%p, 단일 단계 응답은 시간 40% 빠름.

### ⚠️ 제한적 — 새 프로젝트 셋업

쓸 수는 있지만 가성비가 떨어집니다.

- 진단표가 거의 전부 "❌ 없음"이라 변별력이 약함
- 단계 6(진화 원칙)은 비교할 "이전 상태"가 없어서 작동 안 함
- 추천 우선순위가 사실상 "전부 다"가 됨

**새 프로젝트의 권장 워크플로**:

```
[1] Claude Code 내장 /init 명령으로 CLAUDE.md 스캐폴드 생성
        ↓
[2] 2~4주 실사용 — 클로드가 어떤 실수를 하는지 관찰
        ↓
[3] 그때 이 스킬을 처음 호출 — 누적된 실패 패턴이 진단 데이터
```

새 프로젝트에서 이 스킬을 쓰고 싶다면 — 개념 학습용으로 references/ 파일들을 직접 읽고 수동으로 셋업하는 편이 효율적입니다.

### ❌ 부적합

- 단발성 일회 작업 (예: "rm 한 번만 막아줘") — `update-config` 스킬이 더 빠름
- 사용자가 적용까지 자동 처리를 원할 때 — 이 스킬은 명시적으로 "추천만, 적용은 사용자 확정 후" 흐름

## 3. 4가지 사용 모드

스킬은 호출 의도에 따라 자동으로 모드를 결정합니다.

### 모드 A: 전체 흐름

**언제** — 처음 도입할 때, 큰 점검이 필요할 때
**트리거** — "전체", "처음부터", "하네스 봐줘" (모호하면 사용자에게 선택지 제시)

**진행**:
1. Phase 0 — 6단계 진단 표 출력
2. 사용자가 우선순위 선택
3. 선택한 단계부터 5-step 패턴(`a`현재상태 → `b`추천 → `c`확정대기 → `d`적용 → `e`다음 단계)으로 진행

**소요 시간** — 1단계당 5~15분, 6단계 모두 손보면 1~2시간

### 모드 B: 진단만

**언제** — "지금 상태 어떤지만 알고 싶다", 변경은 직접 하겠다
**트리거** — "진단", "상태", "어디 약해", "변경은 하지 말고"

**진행**:
1. Phase 0 진단 출력
2. 우선 보강 추천 (강제력 낮음 → 높음 순)
3. **여기서 종료**

**소요 시간** — 2~5분

이 모드만 정기적으로 돌리는 것도 좋습니다 (월 1회 / 마일스톤 후).

### 모드 C: 특정 단계

**언제** — 손볼 곳이 명확할 때
**트리거** — "단계 3", "hooks만", "rm 막아줘", "테스트 자동화", "code-reviewer 만들어줘"

**진행**:
1. 6단계 표 **생략** — 해당 단계 관련 파일만 짧게 점검
2. 5-step 패턴으로 직행
3. 적용 후 [e] 다음 단계 한 줄 안내

**소요 시간** — 5~10분

iter-2 측정 결과 가장 빠른 모드 (60초 내외).

### 모드 D: 진화 점검

**언제** — 모델 업그레이드 후, 마일스톤 후, 정기 점검
**트리거** — "진화", "재검토", "정리", "모델 올렸어", "오래된 규칙"

**진행**:
1. Phase 0 빠른 진단
2. 진화 트리거 5종 중 어떤 게 발화했는지 분석:
   - 모델 업그레이드 / 마일스톤 종료 / 새 팀원 / 반복 실수 / Hook 누적
3. 사용자에게 4가지 질문:
   - Q1 최근 반복 실수 (추가 후보)
   - Q2 무용해 보이는 규칙 (삭제 후보)
   - Q3 모델 외 다른 트리거
   - Q4 점검 범위
4. 답변 기반 단계별 추천

**소요 시간** — 답변·결정 포함 30~60분

## 4. 시나리오 예제 (자주 쓰는 패턴)

### 예제 1: 처음 도입 — "하네스가 뭔지 모르고 그냥 클로드 쓰고 있었다"

```
사용자: "하네스라는 게 있다던데 내 프로젝트도 봐줘"

→ 모드 A 전체 흐름
→ Phase 0: 대부분 ❌ 또는 ⚠️로 진단
→ 우선순위 추천: 보통 단계 2(deny) → 3(Hook) → 1(CLAUDE.md 보강) 순
→ 사용자가 한 번에 다 할지, 단계별로 할지 선택
```

### 예제 2: 클로드가 자꾸 테스트 안 돌림

```
사용자: "수정 후 npm test 자동으로 돌리게 해"

→ 모드 C 단계 3 직행
→ settings.json + package.json만 점검
→ PostToolUse(Edit|Write|MultiEdit) hook 추천
→ timeout 60s, --bail, --silent 운영 옵션 함께 제시
→ 적용 후 [e] "다음은 단계 2 deny가 자연스럽습니다"
```

### 예제 3: 위험 명령 차단

```
사용자: "rm -rf 같은 거 클로드가 못 쓰게 해줘"

→ 모드 C 단계 2 직행
→ deny 패턴 11개 + ask 4개 추천 (diff 형식)
→ 각 패턴이 막는 시나리오를 표로 설명
→ 적용 후 [e] "다음 우선순위는 단계 3 Hooks"
```

### 예제 4: 모델 업그레이드 후

```
사용자: "Sonnet 4.5에서 Opus 4.7로 올렸어. 하네스 점검 좀"

→ 모드 D 진화 점검
→ Phase 0 빠른 진단
→ 모델 업그레이드의 일반적 함의 + 이 프로젝트 특수성
→ Q1~Q4 질문
→ 답변 기반 잠정 우선순위 (보통 deny 강화 + 잔소리 정리 두 방향)
```

### 예제 5: 정기 진단

```
사용자: "한 달에 한 번 점검 루틴이야. 진단만 해줘"

→ 모드 B 진단만
→ Phase 0 표 + 우선순위만 출력
→ 종료 (다른 단계로 자동 진입 안 함)
```

## 5. 좋은 결과를 얻는 팁

### 진단 전에 — 컨텍스트 제공

스킬은 텍스트 파일만 읽으니, 이런 정보는 사용자가 직접 알려주는 게 효과적:

- "최근 클로드가 한 실수: ..."
- "이 프로젝트는 팀 4명 / 출시까지 2주 남음 / Sonnet 4.5 사용"
- "글로벌 ~/.claude/도 같이 봐줘"

### 추천 받은 후 — 한 번에 다 적용 X

스킬이 추천 4~5개를 나열할 때 "전부 적용"보다 **한 단계씩 — 적용 → 며칠 사용 → 다음 단계** 권장.

이유: deny 정책이 너무 빡빡하거나 Hook이 흐름을 끊으면 바로 알아챌 수 있음. 한꺼번에 다 적용하면 어느 게 문제인지 진단 어려움.

### 적용 후 — 검증 한 사이클

특히 단계 2(deny)와 단계 3(Hook)은 적용 후 의도된 동작 확인:

- deny: 더미 명령(`rm -rf ./tmp_test`)을 클로드에게 시켜 차단되는지 확인
- Hook: 일부러 깨지는 코드 한 줄 넣고 Hook이 실패를 클로드에게 전달하는지 확인

스킬이 적용 후 검증 절차를 안내하지만, 사용자가 "검증 한 번 하자"라고 명시적으로 요청하면 더 꼼꼼해짐.

### 단계 6은 반복 호출

진화 점검은 일회성이 아닙니다. 권장 호출 시점:

- 모델 메이저 업그레이드 후 1주 내
- 큰 마일스톤(v1, 분기 종료) 후
- 분기당 1회 정기

각 호출이 짧으니(30~60분) 부담 없이 반복 가능.

## 6. 안 좋은 결과 / 함정

### 함정 1: 추천을 그대로 다 받아들이기

스킬이 제시하는 deny 패턴 11개, Hook 옵션은 **현재 fixture 가정 기반의 일반론**. 실제 프로젝트에 부적합한 게 섞여있을 수 있음:

- `Bash(sudo:*)` 차단이 막상 인프라 작업할 때 방해
- `Bash(curl:*| sh)` 차단이 정상 셋업 스크립트도 막음
- PostToolUse Hook의 `npm test`가 e2e 포함이면 너무 느림

스킬이 다 추천한다고 다 받지 말고, 자기 프로젝트에 맞춰 **빼거나 좁히는** 결정을 사용자가 해야 함. 스킬이 "패턴 더 추가 / 빼줘" 옵션을 매번 제공하니 활용.

### 함정 2: 글로벌 vs 프로젝트 혼동

`~/.claude/CLAUDE.md`(개인 글로벌)와 `<project>/CLAUDE.md`(팀 공유)는 역할이 다름. 스킬이 명시 안 하면 사용자가 어디에 둘지 명확히 지정:

- 언어 톤, 일반 코딩 습관 → 글로벌
- 이 프로젝트의 빌드 명령, 도메인 규칙 → 프로젝트
- 본인만의 디버그 셋업 → CLAUDE.local.md

### 함정 3: Hook 너무 무겁게

PostToolUse hook은 **편집할 때마다** 실행. 5초 이상 걸리면 흐름 끊고, 결국 사용자가 hook을 끄게 됨.

지키기:
- 빠른 단위 테스트만 자동
- 통합/e2e는 별도 스크립트로
- timeout 명시 (60s 권장)

### 함정 4: CLAUDE.md를 1000줄로 채우기

규칙 추가만 하고 삭제 안 하면 비대해짐. 클로드가 끝까지 안 읽거나 핵심 놓침. 단계 6 진화 점검이 이걸 잡지만, 추가 시점에 "이게 정말 필요한가" 자체검열도 필요.

### 함정 5: 자동 적용 기대

이 스킬은 **명시적으로 "추천만"** 흐름. 사용자가 "yes / 적용해" 라고 말해야만 Edit/Write 들어감. 추천 보고 "왜 안 적용해?" 하지 말고, "적용해"라고 한 번 더 말해야 함.

## 7. 자주 묻는 것

**Q. 평가 결과가 어떻게 나왔나?**
→ `plugins/harness/skills/harness-design-workspace/iteration-2/benchmark.md` 참조. 4 시나리오 100% 통과, 베이스라인 대비 +29.2%p, 단일 단계 응답 시간 −40%.

**Q. 6단계를 다 손볼 필요 있나?**
→ 아니오. 진단 결과 우선순위에 따라 핵심 빈틈만 잡아도 충분. 보통 deny + Hook 두 단계만 잡아도 가장 큰 위험은 막힘.

**Q. 다른 사람의 하네스 베스트 프랙티스를 그대로 복붙하면 안 되나?**
→ 가능은 하지만 권장 X. 하네스는 "현재 모델 + 현재 작업 + 현재 팀"에 맞춘 *가설*이라, 다른 환경의 베스트가 우리 환경에선 발목 잡을 수 있음. 진단부터 시작하는 게 안전.

**Q. references/ 파일을 직접 읽어도 되나?**
→ 네, 학습용으론 좋습니다. 6개 파일이 각각:
- 01-claude-md.md / 02-permissions.md / 03-hooks.md / 04-mcp.md / 05-subagent.md / 06-evolution.md
각 단계의 진단 체크리스트, 추천 패턴, 함정이 정리돼 있음.

**Q. 적용한 것 되돌리고 싶을 때**
→ 모두 텍스트 파일이라 git revert 또는 수동 편집으로 즉시 가능. 스킬이 변경 전 상태를 다시 보여달라고 하면 보여줌.

## 8. 한 줄 요약

> **이미 쓰는 프로젝트의 하네스를 진단해서 약한 부분을 사용자 확정 후 단계적으로 강화하는 도구.**
> 새 프로젝트엔 `/init` + 2~4주 실사용 후 도입이 효율적.
> 추천만 하고 임의로 적용 안 함.
