# 단계 5: AI 분리 — Subagent

> 자기가 만든 결과물을 자기가 평가하면 "잘했다"고 한다. 문제를 발견해놓고도 "별 거 아니야"라고 넘긴다.
> 같은 AI를 비판적으로 만드는 것보다 **검증 전문 AI를 따로 두는 게 쉽다**.

## 이 단계가 잡는 빈틈

- 코드를 만든 클로드가 직접 리뷰 → 본인이 의도한 대로 보임
- 보안/성능/일관성 같은 횡단 관심사를 메인 컨텍스트에서 다루기 어려움
- 큰 작업의 컨텍스트가 너무 길어져서 디테일 놓침
- 검증/탐색 작업이 메인 작업과 섞여 컨텍스트 오염

## 분리의 두 가지 효과

1. **시점 분리** — Generator가 끝낸 결과물을 Evaluator가 신선한 눈으로 봄
2. **컨텍스트 분리** — 서브에이전트는 별도 컨텍스트, 메인은 깔끔하게 유지

```
[Main: Generator]              [Subagent: Evaluator]
"이 기능 만들어"                 "이 변경 리뷰해"
   ↓                               ↑
   코드 작성  → → → → → → → → → → → 결과물
                                    ↓
                              비판적 평가 + 피드백
                                    ↓
[Main: Generator]              ←
"피드백 반영해서 수정"
```

## 진단 체크리스트

`.claude/agents/` 디렉토리 확인:

| 항목 | 양호 ✅ | 약함 ⚠️ | 없음 ❌ |
|------|--------|--------|---------|
| 에이전트 디렉토리 존재 | 있음 | - | 없음 |
| 검증 전용 에이전트 | 있음 (code-reviewer 등) | 만드는 역할만 다수 | 없음 |
| 역할 명세 | description에 트리거/범위 명확 | 모호 | - |
| 모델 분리 | 검증은 더 강한 모델 | 같은 모델 | - |
| 자동 호출 vs 명시적 | 트리거 명확 | 매번 수동 | - |

## 검증 에이전트 추천 카탈로그

가장 자주 쓰는 검증 분리:

### 1. code-reviewer (포괄 리뷰)

**언제**: 큰 변경 후, PR 생성 전
**평가**: 로직 결함, 일관성, 가독성, 명명, 추상화 수준

```markdown
---
name: code-reviewer
description: "코드 변경 리뷰 전문. 메인 클로드가 큰 변경(>50줄, >3파일)을 마쳤을 때 자동 또는 명시적 호출. 로직 결함·일관성·가독성 중심으로 비판적으로 평가하고, 'OK' 남발하지 않는다."
tools: Read, Glob, Grep, Bash
model: opus
---

# 코드 리뷰어

## 역할
방금 작성/수정된 코드를 신선한 눈으로 본다.
목표: 메인 컨텍스트가 놓친 결함 발견.

## 절차
1. 변경 범위 확인 (git diff 또는 명시 파일)
2. 변경 내용 Read — 전체 파일 컨텍스트와 함께
3. 다음 항목으로 평가:
   - 로직 정합성 — edge case, off-by-one, null 처리
   - 일관성 — 주변 코드 스타일/패턴과 맞나
   - 명명 — 의도가 드러나나
   - 추상화 — 너무 얇거나 두껍지 않나
   - 보안 — 인젝션/검증 누락
4. 결과 보고:
   - 🔴 Blocker (반드시 고쳐야)
   - 🟡 Suggestion (고려할 만)
   - 🟢 Praise (유지할 만한 결정)

## 비평 원칙
- "괜찮아 보인다" 같은 무내용 통과 금지 — 구체적 근거 또는 명시적 "찾은 문제 없음"
- 의견과 사실 구분 — "X가 잘못됨"(사실) vs "Y가 더 낫다"(취향)
- 우선순위 매기기 — Blocker가 5개면 가장 중요한 것 먼저
```

### 2. security-reviewer (보안 전담)

**언제**: 인증/인가/입력 검증/외부 API 통신 코드 변경 후
**평가**: OWASP Top 10, 시크릿 노출, 권한 우회

### 3. test-engineer (테스트 전담)

**언제**: 새 기능 후 테스트 부족 의심
**평가**: 커버리지, edge case, mock 적절성, flaky 가능성

### 4. researcher / explorer (탐색 전담)

**언제**: 큰 코드베이스에서 "X가 어디 있지?" 같은 탐색
**효과**: 메인 컨텍스트에 수십 개 파일 내용을 안 끌어와도 됨 — 결과만 받아옴

### 5. debugger (오류 추적 전담)

**언제**: 재현 가능한 버그 또는 명확한 오류 메시지
**평가**: 가설 → 검증 사이클, 근본 원인 vs 증상

## .claude/agents/{name}.md 구조

```markdown
---
name: {kebab-case-name}
description: "(트리거와 범위 명확히. 클로드가 언제 이 에이전트를 호출할지 결정하는 핵심)"
tools: Read, Glob, Grep, Bash, Edit, Write  # 필요한 것만
model: opus  # opus | sonnet | haiku
---

# {에이전트 이름}

## 역할
(1~3줄)

## 절차
(번호 매긴 단계)

## 평가 원칙 / 출력 형식
(에이전트 종류에 따라)
```

원칙:
- **description은 트리거의 핵심** — "리뷰해야 할 때" 같은 추상 표현 X. "큰 변경(>50줄, >3파일) 후" 식으로 구체적
- **tools 최소화** — 검증 에이전트는 보통 Read/Glob/Grep만. Edit/Write는 위험
- **model 선택** — 비판적 평가는 보통 opus가 유리. 단순 탐색은 haiku로 빠르게

## 위치 — `.claude/agents/` 두 곳

| 위치 | 공유 | 용도 |
|------|------|------|
| `<project>/.claude/agents/*.md` | git 커밋, 팀 공유 | 프로젝트 표준 검증 |
| `~/.claude/agents/*.md` | 본인만 | 개인 검증 페르소나 |

프로젝트 특화(예: 이 프로젝트의 ARCHITECTURE.md 준수 검사)는 프로젝트, 일반 검증(예: 커밋 메시지 스타일)은 글로벌.

## 호출 — 자동 vs 명시적

자동 호출은 description의 매칭에 의존한다. 더 확실한 건 사용자/CLAUDE.md에서 명시적 트리거:

```markdown
# CLAUDE.md
## 검증 정책
- 50줄 이상 변경 후에는 반드시 code-reviewer 서브에이전트 호출
- 인증/권한 코드 변경은 security-reviewer 추가 호출
```

또는 사용자가 직접 "code-reviewer로 리뷰해줘"라고 호출.

## 진행 절차 (이 단계 진입 시)

1. 현재 `.claude/agents/`(프로젝트 + 글로벌) 모두 list, 각 .md description Read
2. 진단:
   - 검증 전용 에이전트 유무
   - 만드는 에이전트와 검증하는 에이전트의 분리 정도
3. 사용자에게 묻기: "어떤 검증을 분리하고 싶으세요? 코드 리뷰 / 보안 / 테스트 ..."
4. 추천:
   - 새 에이전트 .md 파일 본문 (위 구조에 맞춰)
   - CLAUDE.md에 호출 정책 추가 (선택)
5. 사용자 확정
6. Write로 `.claude/agents/{name}.md` 생성
7. 적용 후 안내: "이제 다음과 같이 호출 가능 — '{에이전트명}으로 리뷰해줘'"

## 자주 발생하는 함정

- **하나의 에이전트가 다 한다** — 코드도 짜고 리뷰도 하면 분리 효과 사라짐
- **description이 너무 일반적** — "코드 도움" 같은 건 트리거 안 된다. 구체적 상황 명시
- **tools 권한 과다** — 검증자가 Edit/Write 가지면 self-correcting 시도. 위험
- **메인 작업 중간에 매번 호출** — 비싸고 느림. 의미 있는 단위마다 (큰 변경 후, 커밋 전 등)
- **subagent 결과를 무비판적 수용** — Generator가 Evaluator 의견을 비판 없이 다 반영하면 또 다른 over-correction
- **CLAUDE.md만 있고 에이전트 없음** — "이런 점 신경써라" 부탁만으로는 시점 분리/컨텍스트 분리 효과 못 봄
