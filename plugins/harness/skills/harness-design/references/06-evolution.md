# 단계 6: 진화 원칙 (메타)

> 모든 하네스 규칙 = **모델 한계에 대한 가정**.
> 모델이 발전하면 가정이 깨진다. 하네스는 고정된 베스트 프랙티스가 아니라, 현재 모델·현재 작업에 맞춘 가설이다.
> 정기적으로 재검토하지 않으면 하네스가 발목을 잡는다.

## 이 단계가 잡는 빈틈

- 모델 업그레이드(예: Sonnet 4.5 → Opus 4.7) 후에도 옛 모델용 잔소리가 남아있음
- 한 번 추가한 Hook이 더는 필요 없는데 매번 5초씩 잡아먹음
- "Sprint 끝나면 제거"라 적어둔 임시 규칙이 6개월째 살아있음
- 새로 등장한 실패 패턴은 규칙화 안 됨 → 같은 실수 반복

## 진화의 두 방향

```
                       ↑ 규칙 추가
                       │
   [새 실패 발견]   →   하네스 보강
                       │
                       ↓ 규칙 삭제
                       │
   [모델 발전 / 규칙 무용]  →   하네스 정리
```

추가만 하면 비대해지고, 삭제만 하면 다시 실수에 노출된다. **양방향이 같은 빈도로** 움직여야 한다.

## 진화 트리거 — 모델 변경만이 전부가 아니다

진화 점검을 작동시키는 시그널 5가지:

| 트리거 | 빈도 | 어디에 영향 |
|--------|------|------------|
| **모델 업그레이드** (Sonnet 4.5 → Opus 4.7 등) | 1~3개월에 1회 | 잔소리 정리, Hook 간소화, Subagent 통합 검토 |
| **프로젝트 마일스톤 종료** (v1 출시, Sprint 종료) | 마일스톤 단위 | "출시 전까지만" 식 임시 규칙 만료 정리 |
| **새 팀원 합류** | 채용/이동 시 | CLAUDE.md 표준이 새 멤버에게도 통하는지 점검, 암묵 규칙 명문화 |
| **반복 실수 패턴 발견** | 주간 또는 즉시 | CLAUDE.md / Hook / Subagent에 새 규칙 추가 |
| **Hook 실행 누적/지연** (편집할 때마다 5초+) | 모니터링 시 | 무거운 Hook 분리 또는 비활성, 검증 전략 재설계 |

각 트리거가 발화하는 단계가 다르다는 점이 핵심이다:
- 모델 업그레이드 → 주로 **삭제** 방향 (잔소리 → 모델이 자체 처리)
- 마일스톤 종료 → **삭제** (만료 임시 규칙) + **추가** (운영 단계 새 규칙)
- 새 팀원 합류 → **추가** (암묵 → 명문화)
- 반복 실수 → **추가** (실수 → 규칙)
- Hook 누적 → **삭제** 또는 **재설계** (강제력 유지하되 부담 줄이기)

> 한 번에 여러 트리거가 겹치면 (예: 모델 업그레이드 + 마일스톤 종료) 점검을 한 번에 묶는다. 하네스 변경은 텍스트 파일이라 가벼우니까.

## 진단 — 진화 흔적 점검

다음 신호가 있으면 진화 정체 의심:

| 신호 | 의미 |
|------|------|
| CLAUDE.md 마지막 수정일이 6개월+ 전 | 새 실수 규칙화 안 됨 |
| `# TODO 임시` / `# 한시적` 주석이 살아있음 | 만료 규칙 방치 |
| Hook 실행 시간이 길어짐 | 더 이상 필요 없는 검증이 누적 |
| 같은 실수가 매 세션 반복 | 규칙화 누락 |
| 모델 업그레이드 후 1주+ 점검 안 함 | 옛 모델용 잔소리 잔존 |

bash로 확인:

```bash
# CLAUDE.md 최근 수정일
stat -c %y CLAUDE.md 2>/dev/null || stat -f %Sm CLAUDE.md 2>/dev/null

# 임시/만료 표시 검색
grep -rn "TODO 임시\|한시적\|temporary\|REMOVE AFTER\|until" CLAUDE.md .claude/ 2>/dev/null

# Hook 정의 개수
jq '.hooks | to_entries | map(.value | length) | add' .claude/settings.json 2>/dev/null
```

## 진화 점검 절차 — 4단계

이 단계로 진입했을 때 수행:

### 진화-1. 무용 규칙 식별 (삭제 후보)

다음을 차례로 사용자와 확인:

1. **CLAUDE.md 규칙별 검토**
   - 각 규칙에 대해 "이 규칙이 없을 때 모델이 정말 실수하는가?"
   - 옛 모델용("Claude는 TypeScript 잘 못함" 류)은 삭제 후보
   - 만료 표시("v1 출시 후 제거")는 만료됐으면 삭제

2. **Hook 효용 검토**
   - 마지막으로 Hook이 차단/수정한 게 언제인가? 로그 확인 가능하면
   - 같은 검증을 모델이 자발적으로 하고 있는가?

3. **Subagent 사용 빈도**
   - 만든 후 한 번도 호출 안 된 에이전트는 description 부정확 또는 무용

### 진화-2. 새 실패 패턴 → 규칙화 (추가 후보)

최근 작업에서 반복된 실수를 찾는다:

1. 사용자에게 묻기: "최근 N주간 클로드가 자꾸 한 실수가 있나요?"
2. 실수 유형별로 적절한 단계에 배치:
   - "한국어로 답해야 하는데 영어로" → 단계 1 CLAUDE.md
   - "rm -rf을 자꾸 시도" → 단계 2 deny
   - "테스트 안 돌리고 끝냄" → 단계 3 Hook
   - "리뷰 없이 큰 변경 푸시" → 단계 5 Subagent
3. 추천 → 사용자 확정 → 적용

### 진화-3. 모델 변경 영향 검토

사용자에게 모델 변경 이력 묻기. 변경 있었으면:

| 변화 방향 | 검토 항목 |
|----------|----------|
| 더 좋은 모델로 | 자질구레한 잔소리 삭제 가능. Subagent 일부 통합 가능. Hook 일부 무용 |
| 더 가벼운 모델로 | 추가 가이드 필요 가능. 컨텍스트 분리(Subagent) 더 적극적으로 |
| 같은 모델 유지 | 새 실패 패턴 위주 |

### 진화-4. 잔존 임시 규칙 정리

```bash
grep -rn "TODO\|FIXME\|XXX\|임시\|temporary\|REMOVE AFTER\|until\|deadline" \
  CLAUDE.md ~/.claude/CLAUDE.md .claude/ 2>/dev/null
```

각 항목에 대해 "이거 아직 유효한가요?" 확인.

## "Before / After" 사례 (참고)

슬라이드의 사례:
```
Before (Sonnet 4.5)         After (Opus 4.7)
├ Sprint 에이전트            ├ Sprint 제거 ❌
├ Planner 에이전트           ├ Planner 유지 ✓
└ Evaluator 에이전트         └ Evaluator 유지 ✓
```

해석: 모델이 똑똑해져서 짧은 작업의 "스프린트 분할"은 불필요. 하지만 **별도 시점/관점이 필요한** Planner와 Evaluator는 여전히 유효 — 이건 모델 능력이 아니라 *역할 분리*의 가치라서.

이런 식으로 "이 하네스는 모델 한계 때문인가, 역할 분리 때문인가?" 구분하면 진화 결정이 쉬워진다.

## 텍스트 파일이라 즉시 반영

하네스 6요소는 모두 텍스트 파일이다 — `CLAUDE.md`, `settings.json`, `.mcp.json`, `.claude/agents/*.md`. 그래서 진화는 무겁지 않다:

- 한 줄 삭제 = 즉시 반영 (다음 세션부터)
- Hook 비활성 = settings.json 수정만으로 즉시
- 에이전트 정리 = 파일 삭제만

정기 점검 루틴 추천:
- 월 1회 또는 모델 메이저 업그레이드 후
- 큰 프로젝트 마일스톤 후 (만료 규칙 정리 시점)

## 진행 절차 (이 단계 진입 시)

1. 위 진단 신호 점검 (bash 명령 실행)
2. 사용자에게 진화-1~4 묻기:
   - "최근 클로드가 반복한 실수 있나요?" (추가 후보)
   - "현재 규칙 중 더는 필요 없어 보이는 거?" (삭제 후보)
   - "모델 변경했어요?" (재검토 트리거)
   - "임시 규칙 만료된 거 정리할까요?"
3. 식별된 변경마다 추천 → 사용자 확정 → 적용
4. 종료 시 "다음 진화 점검 권장 시점"을 함께 안내:
   - "다음 모델 업그레이드 시"
   - "마일스톤 X 종료 후"
   - "월 1회 정기"

## 자주 발생하는 함정

- **추가만 하고 삭제 안 함** — CLAUDE.md가 1000줄로 비대해짐. 클로드가 끝까지 안 읽음
- **모델 업그레이드 후 점검 누락** — 새 모델이 잔소리에 발목 잡힘 (지나친 검증, 불필요한 분리)
- **"베스트 프랙티스" 추종** — 다른 사람 하네스 그대로 복붙 → 우리 작업과 안 맞음. 가설은 *우리* 환경 기반이어야
- **삭제 결정에 무게 두기** — "혹시 모르니 둬"가 누적되면 비효율. 삭제는 가볍게, 다시 추가도 가벼우니까
- **진화를 별도 일정으로 안 잡음** — 정기성 없이 즉흥적이면 결국 안 함

## 마무리 메시지 (이 단계 종료 시 사용자에게 안내)

> 하네스가 "필요 없어지는" 게 아니라 "필요한 하네스가 바뀌는" 것이다.
> 모델이 좋아질수록 잔소리는 줄지만, 역할 분리·검증 같은 구조적 분업은 오히려 더 가치가 커진다.
> 다음 진화 점검은 {합의된 시점}에 다시 — 그땐 단계 6만 호출해도 충분하다.
