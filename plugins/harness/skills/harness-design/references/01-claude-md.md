# 단계 1: 규칙 전달 — CLAUDE.md

> CLAUDE.md는 매 세션 자동 로드되는 규칙 전달소다.
> 단, **부탁** 수준이다 — 클로드가 안 지킬 수 있다. 강제력이 필요하면 단계 2(Permissions) 또는 단계 3(Hooks)로 보강해야 한다.

## 이 단계가 잡는 빈틈

- 같은 실수를 매 세션 반복 (예: "한국어로 답해" 매번 알려줘야 함)
- 프로젝트 특화 규칙(빌드 명령, 디렉토리 구조)을 매번 재설명
- 코딩 스타일·금지 패턴이 한 곳에 모이지 않아 일관성 무너짐

## 진단 체크리스트

CLAUDE.md를 Read한 뒤 다음 항목으로 평가한다:

| 항목 | 좋음 ✅ | 약함 ⚠️ | 없음 ❌ |
|------|--------|--------|---------|
| 분량 | 적절 (50~300줄) | 과다(500줄+) / 빈약(20줄-) | 파일 자체 없음 |
| 프로젝트 개요 | 1단락에 무엇/왜 명확 | 모호 | 없음 |
| 빌드/실행 명령 | 복붙 가능한 명령 명시 | 산발적 언급 | 없음 |
| 코딩 규칙 | 도메인 특화 규칙 5개 이상 | 일반론만 | 없음 |
| 금지 패턴 | "하지 마라" 명시 | 암묵적 | 없음 |
| 실수→규칙화 흔적 | "이전 X 실수로 추가" 식 주석 | 일반 규칙만 | 없음 |
| 언어/응답 톤 | 명시 | 없음 | 없음 |

## 위치 결정 — 3가지 옵션

| 위치 | 용도 | 공유 |
|------|------|------|
| `~/.claude/CLAUDE.md` | 모든 프로젝트에 공통 적용되는 개인 규칙 | 본인만 |
| `<project>/CLAUDE.md` | 팀이 공유하는 프로젝트 규칙 | 팀 전체 (git에 커밋) |
| `<project>/CLAUDE.local.md` | 팀에 공유하지 않는 개인 메모 | 본인만 (gitignore) |

추천 패턴:
- 언어 톤, 일반 코딩 습관 → 글로벌
- 이 프로젝트의 빌드 명령, 도메인 규칙 → 프로젝트
- 본인만의 디버그 셋업, 임시 메모 → local

## 좋은 규칙의 특징

1. **구체적 행위로 진술** — "잘 짜라"(❌) vs "함수당 50줄 이내, side effect 명시"(✅)
2. **이유 함께** — 왜 이 규칙이 있는지. 없으면 클로드가 예외 상황에서 잘못 판단
3. **금지 못지않게 권장도** — "X 쓰지 마"만 있으면 막막. "X 대신 Y 써라"가 동작 가능
4. **예시 1개** — 추상적 규칙은 예시 한 줄로 명료해진다

## 추천 템플릿 (프로젝트용)

```markdown
# CLAUDE.md

## 프로젝트 개요
{1~2단락. 무엇을 만드는 프로젝트이고 왜 만드는지}

## 빌드 / 실행 / 테스트
\`\`\`bash
# 빌드
{명령}

# 테스트
{명령}

# 개발 서버
{명령}
\`\`\`

## 디렉토리 구조
{핵심만. 자명한 건 빼고}

## 코딩 규칙
- {규칙 1 — 왜 필요한지 한 줄}
- {규칙 2}
- ...

## 금지 패턴
- {하지 말 것 + 대신 무엇을 할지}

## 언어
- 응답: 한국어 / 영어 / ...
- 주석: {규칙}
- 커밋 메시지: {규칙}

## 알려진 실수 → 규칙
이 섹션은 클로드가 했던 실수를 규칙화해서 재발 방지하는 곳.
- {YYYY-MM-DD} {실수 요약} → {규칙}
```

## 분량 관리 — 500줄 넘기면 모듈화

CLAUDE.md가 비대해지면 클로드가 끝까지 안 읽거나 핵심을 놓친다. 두 가지 전략:

**전략 A: 인덱스 + 별도 파일**
```markdown
# CLAUDE.md (인덱스)

자세한 규칙은 다음 파일을 참조한다. 작업 시작 전에 관련 파일을 읽어라.

- 빌드/배포: `.claude/rules/build.md`
- 코딩 규칙: `.claude/rules/coding.md`
- 보안 정책: `.claude/rules/security.md`
```

**전략 B: 트리거 기반 로딩**
"X 작업할 때만 Y 파일 읽어라" 식으로 조건부 지시. 컨텍스트 절약에 유리.

## 진행 절차 (이 단계 진입 시)

1. CLAUDE.md(있으면) 전체 Read
2. 위 진단 체크리스트로 점수화
3. 약한 항목 1~3개 골라서 추천 작성:
   ```
   ## 추천: CLAUDE.md 보강

   ### 변경 1: 빌드 명령 섹션 추가
   사유: 현재 빌드 명령이 명시 안 됨 → 클로드가 매번 추측
   
   ```diff
   ## 디렉토리 구조
   ...
   
   +## 빌드 / 실행
   +\`\`\`bash
   +npm install
   +npm run dev
   +\`\`\`
   ```
   ```
4. 사용자에게 "적용할까요?" 묻기
5. 명시적 승인 후 Edit/Write
6. 적용 후 "{몇} 줄 추가/수정됨. CLAUDE.md 총 {N}줄" 보고

## 자주 발생하는 함정

- **"한 줄 코멘트로 충분하다"** — 문제는 그게 부탁이라는 점. 강제력 필요하면 단계 2/3
- **모든 걸 CLAUDE.md에 몰아넣기** — 500줄 넘으면 집중력 분산. 모듈화 권장
- **언제까지 살아있는 규칙인지 명시 안 함** — "Sprint 끝나면 제거" 같은 임시 규칙은 만료일 적기
- **이전 모델용 규칙 방치** — 단계 6 진화 점검에서 정기적으로 솎아내기
