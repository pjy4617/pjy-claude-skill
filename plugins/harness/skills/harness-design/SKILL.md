# Claude Code 하네스 설계 — 진단·보강·진화

> 하네스(harness)란 클로드가 일하는 환경의 안전장치 + 자동화 + 분업 구조를 말한다.
> 이 스킬은 **이미 쓰고 있는 프로젝트의 하네스를 진단하고, 약한 부분을 단계적으로 강화**하는 워크플로우다.
> 모든 변경은 추천으로 먼저 제시되고, 사용자가 명시적으로 확정한 뒤에만 적용된다.

## 하네스 6가지 구성요소

| # | 구성요소         | 파일/메커니즘                  | 강제력         |
|---|----------------|-------------------------------|---------------|
| 1 | 규칙 전달       | `CLAUDE.md`                   | 부탁 (안 지킬 수도) |
| 2 | 위험 차단       | `settings.json` permissions   | 차단 (실행 자체 불가) |
| 3 | 자동 검증       | `settings.json` hooks         | 강제 확인 (우회 불가) |
| 4 | 테스트 도구     | `.mcp.json` (MCP 서버)         | AI에 도구 부여      |
| 5 | AI 분리         | `.claude/agents/` (Subagent)  | 만드는 AI ≠ 검증 AI |
| 6 | 진화 원칙       | (메타) 정기 재검토              | 모델 발전에 따라 변경 |

핵심 인식: 같은 "규칙"이라도 어디에 적어두느냐에 따라 강제력이 다르다. CLAUDE.md에 "테스트 꼭 돌려"라고 적는 것과, Hook으로 강제 실행시키는 것은 전혀 다른 결과를 만든다.

## 진행 모드 결정

`$ARGUMENTS`로 모드를 받는다. 비어있으면 사용자에게 묻는다:

| 모드            | 트리거                     | 동작                                        |
|----------------|--------------------------|---------------------------------------------|
| **전체 흐름**     | 빈 인자, "전체", "처음부터" | Phase 0 전체 진단(6단계 표) → 1~6 순차       |
| **진단만**        | "진단", "상태", "어디 약해" | Phase 0 전체 진단(6단계 표) — 보고서만 출력  |
| **특정 단계**     | "단계 3", "hooks만", "rm 막아줘" | **간이 진단(해당 단계 관련 파일만)** → 해당 단계만 |
| **진화 점검**     | "진화", "재검토", "정리", "모델 올렸어" | Phase 0 + Phase 6                |

모드가 모호하면 사용자에게 "전체 흐름으로 갈까요, 특정 단계만 보겠어요?"를 한 번 묻고 진행한다.

### 모드별 진단 깊이

- **전체 흐름 / 진단만 / 진화 점검** → 6단계 전부 표 형식으로 ✅/⚠️/❌ 진단 (Phase 0 전체)
- **특정 단계** → 해당 단계 관련 파일만 Read하고 한두 줄 요약. 6단계 표를 매번 출력하지 않는다. 예: "rm 막아줘"면 settings.json만 보고 deny 정책 추천으로 직행. 사용자가 "전체 진단도 해줘"라고 추가 요청하지 않는 한, 6단계 표는 생략.

이유: 단일 단계 요청에 매번 전체 진단표를 출력하면 토큰을 낭비하고 사용자가 원하는 답까지의 거리가 멀어진다.

## Phase 0: 진단 (모든 모드의 시작)

### 0-1. 파일 존재 확인

```bash
# 프로젝트 루트에서 실행
ls -la .claude/ 2>/dev/null
ls -la CLAUDE.md AGENTS.md CLAUDE.local.md 2>/dev/null
ls -la .mcp.json .claude/.mcp.json 2>/dev/null
ls -la .claude/settings.json .claude/settings.local.json 2>/dev/null
ls -la .claude/agents/ 2>/dev/null
ls -la .claude/hooks/ 2>/dev/null
```

### 0-2. 글로벌 하네스도 확인

사용자 글로벌 설정도 영향을 주므로 함께 본다:

```bash
ls -la ~/.claude/CLAUDE.md ~/.claude/settings.json ~/.claude/agents/ 2>/dev/null
```

(Windows: `~/.claude` ≒ `%USERPROFILE%\.claude`)

### 0-3. 내용 요약 (있는 파일만)

각 파일을 Read로 열어 **핵심만 요약**한다 — 파일 전체를 그대로 출력하지 않는다.

- `CLAUDE.md`: 줄 수, 주요 섹션 헤더, "이전 실수에서 배운 규칙" 흔적 유무
- `settings.json` (project & local): permissions allow/deny 개수, hooks 정의 유무
- `.mcp.json`: 등록된 MCP 서버 이름 목록
- `.claude/agents/`: 에이전트 파일명 목록
- `.claude/hooks/`: 스크립트 파일명 목록

### 0-4. 진단 보고서 (출력 형식)

다음 표를 사용자에게 보여준다:

```markdown
## 하네스 진단 결과

| 단계 | 구성요소            | 상태       | 비고                       |
|-----|--------------------|----------|---------------------------|
| 1   | CLAUDE.md          | ✅ 양호   | 120줄, 코드 규칙 7개        |
| 2   | Permissions        | ⚠️ 약함   | allow만 있고 deny 없음     |
| 3   | Hooks              | ❌ 없음   | 자동 검증 미설정            |
| 4   | MCP                | ❌ 없음   | 테스트 도구 연결 없음        |
| 5   | Subagent           | ⚠️ 부분   | 1개만 — 검증 전문 부재     |
| 6   | 진화 흔적          | ⚠️ 정체   | 최근 6개월 수정 이력 없음   |

### 우선 보강 추천 (강제력 낮음 → 높음 순)
1. **단계 3 Hooks** — 가장 큰 빈틈. 현재 "테스트 돌려라"가 부탁 수준
2. **단계 2 deny** — `Bash(rm -rf *)` 같은 위험 명령 차단
3. **단계 5 검증 AI** — code-reviewer 서브에이전트 추가
```

상태 표시:
- ✅ 양호 — 의미 있는 내용 존재
- ⚠️ 약함/부분 — 있긴 하나 빈약하거나 한쪽만 있음
- ❌ 없음 — 파일 자체가 없음

### 0-5. 다음 단계 결정

진단 후 사용자에게 묻는다:
- "어디부터 손볼까요? 우선순위 추천은 위와 같습니다."
- 답변에 따라 Phase 1~6 중 선택 진행
- "진단만"이면 여기서 종료

## Phase 1~6: 각 단계 진행 패턴

각 단계는 동일한 5-step 패턴으로 진행한다:

```
[a] 현재 상태 재확인 (해당 단계 관련 파일만 자세히)
 ↓
[b] 변경 추천 제시 (diff 형식, 왜 이렇게 바꾸는지 함께)
 ↓
[c] 사용자 확정 대기 — "적용할까요?"
 ↓
[d] 명시적 승인(예/yes/적용) 후에만 Write/Edit
 ↓
[e] 적용 후 다음 단계 한 줄 안내 — 단계 간 의존성·우선순위 기반
```

승인되지 않은 추천은 절대 적용하지 않는다. 이건 협상 가능한 규칙이 아니다 — 하네스 자체가 사용자 통제 도구이기 때문에, 도구가 사용자 동의 없이 환경을 바꾸면 신뢰가 무너진다.

[e]의 다음 단계 안내 예시:
- 단계 2(deny) 적용 후 → "다음 우선순위는 단계 3(Hooks)입니다. deny 정책이 정해졌으니 이제 PostToolUse hook을 안전하게 추가할 수 있어요. 진행할까요?"
- 단계 1(CLAUDE.md) 적용 후 → "CLAUDE.md가 정리됐으니 단계 5(Subagent)에서 검증 전문 AI를 추가하면 시너지 큽니다. 보시겠어요?"
- 단계 3(Hooks) 적용 후 → "Hook이 강제 검증을 가져갔으니 다음은 단계 5(Subagent)나 단계 1(CLAUDE.md 보강) 중 선택. 어디로 갈까요?"

다음 단계가 없거나 사용자가 "여기까지" 같은 종료 신호를 주면 [e]를 생략한다.

각 단계의 상세 패턴, 진단 체크리스트, 추천 템플릿은 references/에 있다:

| 단계 | 주제              | 참조 파일                    |
|-----|-------------------|----------------------------|
| 1   | 규칙 전달         | `references/01-claude-md.md` |
| 2   | 위험 차단         | `references/02-permissions.md` |
| 3   | 자동 검증         | `references/03-hooks.md` |
| 4   | 테스트 도구       | `references/04-mcp.md` |
| 5   | AI 분리           | `references/05-subagent.md` |
| 6   | 진화 원칙         | `references/06-evolution.md` |

해당 단계로 넘어가기 직전에 **그 단계의 reference 파일을 Read**한 다음 진행한다. 모든 reference를 미리 읽지 않는다 — 컨텍스트 절약과 단계별 집중을 위해서다.

## 단계 간 의존성과 권장 순서

기본 순서는 1 → 2 → 3 → 4 → 5 → 6이지만, 의존성이 있다:

- **Hooks (3)는 Permissions (2)보다 뒤** — Hook이 실행하는 명령 자체가 deny에 걸리면 안 되니, deny 정책을 먼저 정해야 한다
- **Subagent (5)는 CLAUDE.md (1)보다 뒤** — 서브에이전트도 CLAUDE.md를 읽으므로, 메인 규칙이 먼저 정리돼야 한다
- **MCP (4)는 독립적** — 다른 단계와 무관하게 추가 가능

진단 결과 우선순위가 다르면 순서를 조정한다. 예: 위험한 명령이 자주 실행되는 환경이면 단계 2를 먼저.

## 핵심 원칙: 추천만, 적용은 사용자 확정 후

이 스킬은 다음을 **하지 않는다**:
- 사용자 동의 없이 `CLAUDE.md`, `settings.json`, `.mcp.json`을 생성/수정
- "이게 표준이니까 적용했어요" 식의 일방 통보
- 추천 없이 곧장 Write로 넘어가기

이 스킬은 다음을 **한다**:
- 현재 상태 진단 → 표 형식 보고
- 단계별 추천 → diff 형식 + "왜 이렇게 바꾸는지" 설명
- 사용자가 "적용해" / "yes" / "그렇게 해줘" 등 명시적 승인 후 Write/Edit
- 적용 후 결과 확인 (어떤 파일 어떻게 바뀌었는지 짧게 요약)

승인이 모호한 경우(예: "음... 그럴 수도?")는 **승인 아님**으로 처리하고 다시 묻는다.

---

## 진화의 의미 — 단계 6의 무게

슬라이드의 메타 원칙: **모든 하네스 규칙 = 모델 한계에 대한 가정**.

- 옛 모델이 자꾸 빠뜨리던 일을 강제 Hook으로 잡았다면, 새 모델에선 그 Hook이 불필요할 수 있다
- 옛 모델이 못 하던 작업을 Subagent로 분리했다면, 새 모델에선 한 곳에서 다 할 수 있을 수도 있다
- 거꾸로, 새 모델이 더 적극적으로 파일을 건드린다면 deny가 더 빡빡해져야 할 수도 있다

하네스는 **고정된 베스트 프랙티스가 아니라 현재 모델과 현재 작업에 맞춘 가설**이다. 단계 6은 이 가설을 정기적으로 재검토하는 절차다. 자세한 내용은 `references/06-evolution.md`.

## 사용 예시

```
사용자: "내 프로젝트 하네스 좀 봐줘"
→ 전체 흐름: 진단 → 우선순위 → 단계별 추천

사용자: "클로드가 자꾸 npm test를 안 돌려"
→ 단계 3 Hooks로 직행: PostToolUse(Edit) → npm test 자동 실행 추천

사용자: "rm -rf 같은 거 막아줘"
→ 단계 2 Permissions로 직행: deny 정책 추천

사용자: "Opus 4.7로 올렸는데 하네스 그대로 둬도 돼?"
→ 단계 6 진화 점검: 안 쓰는 규칙, 시대에 뒤처진 Hook 식별

사용자: "지금 상태만 알려줘"
→ Phase 0 진단 보고서만 출력하고 종료
```

$ARGUMENTS
