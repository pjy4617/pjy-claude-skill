## 분석 — 어디서 차이가 나는가

### 큰 차이 — eval-1 (full-diagnosis), eval-4 (evolution-check)
스킬의 **프레임워크 자체가 가치**인 시나리오. 베이스라인은:
- eval-1: 6단계 표준 분류(특히 MCP, 진화 흔적)를 안 씀, 일반적인 코드 리뷰처럼 응답
- eval-4: "단계 6 진화 점검" 개념이 없어서, 모델 업그레이드를 단순 "보강할 거리"로만 봄. **삭제/추가 양방향**, "모든 규칙은 모델 한계 가정" 같은 메타 관점 부재
- 베이스라인이 던지는 질문도 "지금 보강할까요?" 식이라 진단 → 우선순위 → 사용자 선택의 라우팅 흐름을 따라가지 않음

### 차이 미미 — eval-2 (permissions)
"rm 막아줘"는 일반적인 요청이라 베이스라인도 deny 정책을 바로 추천. 스킬과 베이스라인 모두 6/6. 다만 스킬은 단계 명시 + 단계 간 의존성(deny 정책이 Hook보다 먼저) 같은 **컨텍스트화**가 추가됨.

### 작은 차이 — eval-3 (hooks)
베이스라인도 PostToolUse Hook 추천은 잘 함. 다만 **timeout / exit 코드** 같은 운영 고려사항을 놓침 — 스킬은 references/03-hooks.md에 5원칙으로 박아뒀기 때문에 100%로 챙김.

## 비용 분석

with_skill이 토큰을 평균 **+9,598 (28%) 더** 씀 — references/ 파일 로딩 비용. 시간은 거의 동일.

품질 향상폭(+29.2%p) 대비 토큰 28% 증가는 합리적. 특히 풀 진단·진화 같은 "구조적 사고가 필요한" 시나리오에서 가성비 좋음.

## 베이스라인이 강한 영역에서 배운 점

eval-2(permissions)에서 베이스라인이 동률을 보임 → 단순한 단일 단계 요청에서는 스킬이 **순수 가치 더해주지 못함**. 스킬의 진짜 가치는:
- 다단계 진단 (eval-1)
- 메타 관점 (eval-4)
- 단계 간 의존성·일관성 (eval-3의 운영 고려사항)

그 외 단순 "X 막아줘" 류는 일반 모델로도 충분.

## 종합 평가

**Iteration 1 결과: 통과** — 4개 시나리오 모두 100%. 베이스라인 대비 +29.2%p, 특히 다단계/메타 시나리오에서 +50~60%p의 큰 차이.

다음 손볼 만한 곳:
1. eval-2 같은 단순 라우팅 시나리오에서 토큰을 줄이는 방향 — 스킬이 진단을 "간략 한 줄"로 끝내고 본 단계로 직행하는 흐름이 잘 됐음 (with_skill 응답 보면 Phase 0 표를 짧게만 출력함). 추가 최적화는 마이너.
2. references/06-evolution.md에 "Before/After" 사례 더 다양화 — 사용자가 모델 변경 외 다른 진화 트리거(프로젝트 마일스톤, 새 팀원 합류 등)를 주면 어떻게 응답할지
