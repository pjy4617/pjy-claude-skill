# harness-design — Iteration 2 Benchmark

> Iteration 1 → 2 비교. 베이스라인(without_skill)은 iteration-1 데이터를 재사용 (동일 모델·동일 프롬프트라 변하지 않음).

## 적용한 개선 3가지

1. **SKILL.md** — 모드별 진단 깊이 명시. "특정 단계" 모드는 6단계 표를 매번 출력하지 않고 해당 단계 관련 파일만 짧게 진단
2. **SKILL.md** — 단계 진행 패턴을 4-step → 5-step으로 확장. [e] "적용 후 다음 단계 한 줄 안내" 추가, 단계 간 의존성 기반 라우팅 명시
3. **references/06-evolution.md** — 진화 트리거 5종 추가 (모델 업그레이드 / 마일스톤 종료 / 새 팀원 / 반복 실수 / Hook 누적·지연). 각 트리거가 발화하는 단계를 표로 매핑

## 정량 점수는 동률, 질적 개선은 명확

iter-1이 이미 with_skill 100%였으므로 어설션 통과율은 변하지 않음. **개선의 효과는 시간(−35%)과 응답 구조에서 드러남**:

### 개선 1 검증 — 모드별 진단 깊이
- iter-1 eval-2 응답: 6단계 표 출력 후 단계 2로 진입 (불필요한 진단)
- iter-2 eval-2 응답: 단계 2 관련 파일(`settings.json`)만 직접 점검 → "[a] 현재 상태"가 표 1개로 끝남
- 결과: 단일 단계 요청에서 시간 40% 절감 (98s → 59s)

### 개선 2 검증 — [e] 다음 단계 안내
- iter-1 응답들: 자체 판단으로 종종 다음 단계를 언급 (일관성 부족)
- iter-2 응답들: 모든 단일 단계 응답에 "[e] 다음 단계 예고/미리보기" 섹션이 일관되게 등장
- 결과: 단계 간 의존성(deny → Hook → CLAUDE.md → Subagent)이 사용자에게 더 명확히 전달

### 개선 3 검증 — 진화 트리거 5종
- iter-1 eval-4 응답: Q1~Q4가 모델 업그레이드 중심 (Q3=직전 모델, Q4=새 우려)
- iter-2 eval-4 응답: Q3가 **"모델 변경 외 다른 트리거 — 마일스톤/새 팀원/Hook 누적"**으로 확장. 비-모델 진화 시그널을 사용자가 한 번에 보고 결정
- 결과: 진화 점검의 적용 범위가 넓어짐 (모델 업그레이드 1종 → 5종)

## 비용

- 시간: **35% 절약** (105.7s → 68.3s) — 가장 큰 실질적 이득
- 토큰: 거의 동일 (+1.4%) — 본문이 약간 늘었으나 무시할 수준
- 품질: 동률 (24/24 → 24/24) + 질적 개선

## iteration-1 → iteration-2 평가

**개선 성공.** 정량 점수는 천장(100%)이라 변동 없지만:
- 응답 구조가 SKILL.md 명세와 더 정확히 일치
- 단일 단계 요청에서 35~40% 빠른 응답
- 진화 점검이 모델 업그레이드 외 트리거까지 커버

추가 iteration이 의미 있는 시점은 — 새 시나리오를 추가해서 천장을 깰 때. 현재 4개 시나리오로는 더 측정할 게 적음.
