# 테스트벤치 코드 리뷰 체크리스트

> 이 스킬은 체크리스트를 정의합니다.
> 실제 리뷰 작업은 tb-reviewer 에이전트에게 위임됩니다.
> RTL 리뷰(rtl-review)와 관점이 다릅니다:
>   - RTL 리뷰 → "합성 가능한가?" (하드웨어 관점)
>   - TB 리뷰  → "검증이 충분한가?" (검증 관점)

## 리뷰 범위

리뷰 요청 시 아래 6개 카테고리를 **모두** 검사합니다.
각 항목은 PASS / WARN / FAIL로 판정합니다.

---

## 카테고리 1: 구조 & 기본 요소 (Structure)

테스트벤치의 기본 골격이 갖춰져 있는지.

| # | 체크 항목 | FAIL 조건 |
|---|----------|-----------|
| ST1 | 파일명 컨벤션 | `tb_모듈명.v` 형식 미준수 |
| ST2 | `timescale 지시자 | `timescale 1ns / 1ps` 누락 |
| ST3 | DUT 인스턴스 | 테스트 대상 모듈 인스턴스 없음 |
| ST4 | 클럭 생성 | `always #(CLK_PERIOD/2) clk = ~clk;` 패턴 없음 |
| ST5 | 리셋 시퀀스 | rst_n을 0으로 시작 → 일정 시간 후 1로 해제하는 코드 없음 |
| ST6 | `$finish` 보장 | 시뮬레이션 종료 보장 메커니즘 없음 |
| ST7 | 타임아웃 워치독 | 무한 루프 방지용 타임아웃 없음 |
| ST8 | 파형 덤프 | `$dumpfile` / `$dumpvars` 누락 (GUI 디버깅 불가) |

## 카테고리 2: 자기 검증 (Self-Checking)

사람이 파형을 눈으로 확인하지 않아도 PASS/FAIL을 판단할 수 있는지.
**자동화된 검증 없이 파형만 의존하는 테스트벤치는 FAIL입니다.**

| # | 체크 항목 | FAIL 조건 |
|---|----------|-----------|
| SC1 | 자동 PASS/FAIL 판정 | `$display("PASS")`/`$display("FAIL")` 없음 |
| SC2 | 기대값 비교 | 출력값과 기대값의 명시적 비교(`===`, `!==`) 없음 |
| SC3 | 테스트 카운터 | pass_count / fail_count 집계 없음 |
| SC4 | 최종 요약 | 시뮬레이션 끝에 전체 결과 요약 출력 없음 |
| SC5 | 에러 메시지 구체성 | FAIL 시 어떤 값이 기대되었고 실제로 뭐였는지 출력 안 함 |

## 카테고리 3: 테스트 커버리지 (Coverage)

DUT의 기능을 얼마나 빠짐없이 검증하는지.

| # | 체크 항목 | WARN 조건 |
|---|----------|-----------|
| CV1 | 정상 동작 (happy path) | 기본 기능 테스트 없음 |
| CV2 | 경계값 테스트 | 입력 최소값(0x00), 최대값(0xFF) 미테스트 |
| CV3 | 리셋 동작 | 리셋 중 동작, 리셋 해제 후 복귀 미검증 |
| CV4 | 연속 동작 | 연속 2회 이상 동작(back-to-back) 미테스트 |
| CV5 | 동시 입력 | 여러 입력이 동시에 변할 때 동작 미테스트 |
| CV6 | FSM 전이 커버리지 | FSM이 있으면 모든 상태 전이를 거치는지 확인 |
| CV7 | 에러/예외 조건 | 비정상 입력(valid 없이 data 변경 등) 미테스트 |
| CV8 | 프로토콜 타이밍 | 프로토콜이 있으면(SPI, UART 등) 비트 타이밍 검증 |

## 카테고리 4: DUT-TB 정합성 (DUT Matching)

테스트벤치가 DUT의 실제 인터페이스와 정확히 맞는지.

| # | 체크 항목 | FAIL 조건 |
|---|----------|-----------|
| DM1 | 포트 완전 연결 | DUT의 모든 포트가 TB에서 연결되었는지 |
| DM2 | 포트 폭 일치 | TB의 reg/wire 폭과 DUT 포트 폭 불일치 |
| DM3 | 포트 방향 일치 | DUT input → TB에서 reg, DUT output → TB에서 wire |
| DM4 | 파라미터 오버라이드 | 시뮬레이션용 파라미터(예: 짧은 딜레이)가 적절한지 |
| DM5 | 미사용 출력 모니터링 | DUT 출력이 있는데 TB에서 아무 검사도 안 함 |

## 카테고리 5: 스타일 & 유지보수 (Style)

테스트벤치의 가독성과 유지보수 용이성.

| # | 체크 항목 | WARN 조건 |
|---|----------|-----------|
| SY1 | 테스트 케이스 분리 | 모든 테스트가 하나의 initial 블록에 뭉쳐있음 |
| SY2 | task 활용 | 반복 패턴(send_byte, check_result 등)을 task로 추출 안 함 |
| SY3 | 테스트 케이스 라벨 | 각 테스트에 `$display("--- Test N: 설명 ---")` 없음 |
| SY4 | 매직 넘버 | 하드코딩된 숫자 (딜레이, 데이터) |
| SY5 | 주석 | 테스트 의도/목적 설명 누락 |
| SY6 | 시뮬레이션 시간 | 불필요하게 긴 시뮬레이션 시간 (최적화 가능) |

## 카테고리 6: 기능별 전문 체크 (Domain-Specific)

프로토콜이나 특정 기능에 대한 전문 검증.

### SPI 인터페이스
| # | 체크 항목 |
|---|----------|
| DS1 | SCLK 주파수가 스펙 범위 내인지 |
| DS2 | CPOL/CPHA 모드가 올바른지 |
| DS3 | CS(SS) 신호 타이밍이 정확한지 |
| DS4 | MSB/LSB first 순서가 올바른지 |

### UART 인터페이스
| # | 체크 항목 |
|---|----------|
| DS5 | Baud rate가 정확한지 (비트 주기 검증) |
| DS6 | Start bit / Stop bit 검증 |
| DS7 | 프레이밍 에러 시나리오 테스트 |

### FSM (범용)
| # | 체크 항목 |
|---|----------|
| DS8 | 모든 상태에 진입하는 테스트 존재 |
| DS9 | 불법 상태 복구 테스트 |
| DS10 | 상태 전이 간 타이밍 검증 |

### 프로젝트별 전문 체크 (예시 — 필요 시 추가)

프로젝트에 따라 아래와 같은 항목을 추가합니다.
이 항목들은 기본 체크리스트에 포함되지 않으며, 프로젝트 특성에 맞게 직접 편집하세요.

**OLED 디스플레이 프로젝트 예시:**
| # | 체크 항목 |
|---|----------|
| PJ1 | 초기화 시퀀스 완전성 (VDD→delay→commands→VBAT→delay→display ON) |
| PJ2 | 전원 OFF 시퀀스 (display OFF→VBAT OFF→delay→VDD OFF) |
| PJ3 | 쿨다운 타이밍 준수 (3.5초) |

**모터 드라이버 프로젝트 예시:**
| # | 체크 항목 |
|---|----------|
| PJ1 | PWM 듀티사이클 경계값 (0%, 100%) 테스트 |
| PJ2 | 과전류 보호 트리거 테스트 |
| PJ3 | 방향 전환 시 데드타임 검증 |

## 리포트 형식

```
═══════════════════════════════════════
  테스트벤치 리뷰 리포트
  대상 TB: [테스트벤치 파일명]
  대상 DUT: [DUT 모듈명]
═══════════════════════════════════════

[요약]
  PASS: 22 / WARN: 4 / FAIL: 2 / TOTAL: 28
  추정 기능 커버리지: ~65% (CV6, CV7 미충족)

[FAIL 항목] (반드시 수정)
  ❌ SC1 — tb/tb_oled_ctrl.v
     자동 PASS/FAIL 판정 없음. 파형 수동 확인에 의존.
     수정: check task 추가, 기대값 비교 로직 구현

  ❌ DM1 — tb/tb_spi_ctrl.v
     DUT 출력 'busy' 포트가 TB에서 미연결
     수정: wire busy; 선언 후 .busy(busy) 연결, 
           전송 중 busy=1 검증 추가

[WARN 항목] (권장 수정)
  ⚠️ CV2 — 경계값 0x00, 0xFF 미테스트
  ⚠️ CV4 — 연속 전송(back-to-back) 미테스트
  ⚠️ CV6 — FSM 상태 3/5만 커버 (S_IDLE, S_INIT, S_READY)
           미커버: S_SEND_CMD, S_DELAY
  ⚠️ SY4 — 딜레이 100 하드코딩 → parameter DELAY = 100 권장

[커버리지 개선 제안]
  다음 테스트 케이스를 추가하면 커버리지가 ~65% → ~90%로 개선:
  1. S_SEND_CMD 상태 진입 (command byte 전송)
  2. S_DELAY 상태 진입 (전원 시퀀스 딜레이)
  3. 경계값 0x00, 0xFF 전송
  4. back-to-back 연속 전송
```

## Vivado 시뮬레이션 결과와의 연계

이 리뷰는 시뮬레이션 실행 전에 수행합니다.
시뮬레이션 후 FAIL이 나왔을 때 원인을 찾기 위해 다시 리뷰할 수도 있습니다.

| 시뮬레이션 결과 | TB 리뷰 관점 |
|---------------|-------------|
| PASS | 커버리지가 충분한지 다시 확인 (쉬운 케이스만 통과한 건 아닌지) |
| FAIL | FAIL 원인이 DUT 버그인지 TB 버그인지 구분 |
| 타임아웃 | ST6/ST7 체크 — $finish 누락 또는 워치독 부재 |
| 파형 없음 | ST8 체크 — $dumpfile 누락 |
