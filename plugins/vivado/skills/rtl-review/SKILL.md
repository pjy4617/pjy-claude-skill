# Verilog RTL 코드 리뷰 체크리스트

> 이 스킬은 체크리스트를 정의합니다.
> 실제 리뷰 작업은 rtl-reviewer 에이전트에게 위임됩니다.
> 에이전트는 별도 컨텍스트에서 모든 RTL 파일을 읽고 이 체크리스트를 적용합니다.

## 리뷰 범위

리뷰 요청 시 아래 5개 카테고리를 **모두** 검사합니다.
각 항목은 PASS / WARN / FAIL로 판정합니다.

---

## 카테고리 1: 합성 가능성 (Synthesizability)

가장 중요. 합성 불가능한 코드는 Vivado에서 에러 또는 의도치 않은 래치를 생성합니다.

| # | 체크 항목 | FAIL 조건 | 자동 수정 가능? |
|---|----------|-----------|---------------|
| S1 | 래치 추론 없음 | `if` 에 `else` 없음, `case`에 `default` 없음 | rtl-designer에게 수정 요청: else/default 추가 |
| S2 | 멀티 드라이버 없음 | 같은 신호를 여러 always 블록에서 구동 | ❌ 설계 재검토 필요 |
| S3 | 조합 루프 없음 | assign A = f(B), assign B = g(A) | ❌ 설계 재검토 필요 |
| S4 | 초기화되지 않은 reg 없음 | reg 선언 후 리셋 블록에서 초기화 누락 | rtl-designer에게 수정 요청: 리셋 블록에 추가 |
| S5 | 비동기 리셋 확인 | `always @(posedge clk or negedge rst_n)` | ⚠️ WARN |
| | | Xilinx FPGA에서 비동기 리셋도 합성 가능하나, 동기식 권장 | |
| S6 | `initial` 블록 미사용 (합성용) | 합성 대상 모듈에서 `initial` 사용 | rtl-designer에게 수정 요청: 리셋 로직으로 변경 |
| S7 | `assign` 최소화 | 복잡한 조합 로직을 assign으로 작성 | ⚠️ always 블록으로 권장 |

## 카테고리 2: 코딩 스타일 (Coding Style)

CLAUDE.md 및 rtl-designer 에이전트의 규칙 준수 여부.

| # | 체크 항목 | WARN 조건 |
|---|----------|-----------|
| C1 | 모듈명 snake_case | `SpiCtrl` → `spi_ctrl` |
| C2 | 클럭/리셋 컨벤션 | `clk`, `rst_n` (active low) 미준수 |
| C3 | 레지스터 `_r` 접미사 | 레지스터 출력에 `_r` 없음 |
| C4 | 와이어 `_w` 접미사 | 조합 신호에 `_w` 없음 |
| C5 | FSM 3-process 스타일 | state_reg, next_state, output 미분리 |
| C6 | 파라미터화 | 매직 넘버 하드코딩 (예: `868` 대신 `CLKS_PER_BIT`) |
| C7 | 주석 | 모듈 헤더, FSM 상태 설명 누락 |

## 카테고리 3: 포트/계층 정합성 (Port & Hierarchy)

모듈 간 연결이 올바른지 교차 검증. **여러 파일을 동시에 분석해야 함.**

| # | 체크 항목 | FAIL 조건 |
|---|----------|-----------|
| H1 | 포트 폭 일치 | 인스턴스의 포트 폭과 선언된 wire/reg 폭 불일치 |
| H2 | 미연결 포트 없음 | 인스턴스에서 포트를 비워둠 (`.port_name()`) |
| H3 | 미사용 포트 없음 | 선언되었지만 어디에도 연결되지 않은 포트 |
| H4 | Top 모듈 포트 ↔ XDC 일치 | XDC에 정의된 포트가 top.v에 없거나, 반대의 경우 |
| H5 | 클럭 도메인 일관성 | 모든 모듈이 같은 clk을 사용하는지 (CDC 없으면) |

## 카테고리 4: 타이밍 & 리소스 (Timing & Resource)

합성 전 예측 가능한 타이밍/리소스 이슈.

| # | 체크 항목 | WARN 조건 |
|---|----------|-----------|
| T1 | 깊은 조합 로직 | case문 중첩 3단 이상, 긴 if-else 체인 |
| T2 | 큰 MUX | 32:1 이상의 MUX 추론 가능성 |
| T3 | 카운터 크기 | 32비트 이상 카운터 (타이밍 영향) |
| T4 | 메모리 추론 | 대형 reg 배열 → BRAM으로 추론되는지 확인 |
| T5 | 클럭 이네이블 | 게이티드 클럭 대신 클럭 이네이블 사용 권장 |

## 카테고리 5: 테스트벤치 품질 (Testbench Quality)

시뮬레이션 커버리지 및 테스트벤치 완성도.

| # | 체크 항목 | WARN 조건 |
|---|----------|-----------|
| B1 | 테스트벤치 존재 | 모듈에 대응하는 tb_모듈명.v 없음 |
| B2 | 자동 검증 | `$display` PASS/FAIL 없이 파형만 의존 |
| B3 | 리셋 테스트 | 리셋 동작 검증 케이스 없음 |
| B4 | 엣지 케이스 | 경계값(0x00, 0xFF), 오버플로우 테스트 없음 |
| B5 | 타임아웃 워치독 | `$finish` 보장 메커니즘 없음 |
| B6 | 파형 덤프 | `$dumpfile` / `$dumpvars` 없음 |

## 리포트 형식

리뷰 결과는 아래 형식으로 출력합니다:

```
═══════════════════════════════════════
  RTL 코드 리뷰 리포트
  프로젝트: [프로젝트명]
  리뷰 대상: [파일 목록]
  리뷰 일시: [날짜]
═══════════════════════════════════════

[요약]
  PASS: 28 / WARN: 3 / FAIL: 1 / TOTAL: 32

[FAIL 항목] (반드시 수정)
  ❌ S1 — rtl/oled_ctrl.v:87
     case문에 default 없음 → 래치 추론 위험
     수정: default: next_state_w = S_IDLE; 추가

[WARN 항목] (권장 수정)
  ⚠️ C6 — rtl/spi_ctrl.v:23
     매직 넘버 50 → 파라미터 SPI_CLK_DIV로 변경 권장
  ⚠️ B4 — tb/tb_spi_ctrl.v
     경계값 테스트 없음 → 0x00, 0xFF 케이스 추가 권장
  ⚠️ T1 — rtl/oled_ctrl.v:120
     case문 4단 중첩 → 파이프라인 분리 고려

[PASS 항목] (이상 없음)
  ✅ S2~S7, C1~C5, C7, H1~H5, T2~T5, B1~B3, B5~B6
```

## Vivado 린트와의 차이

이 리뷰는 Vivado 합성 전에 수행합니다.
Vivado가 잡지 못하는 항목(코딩 스타일, 계층 정합성, 테스트벤치 품질)을 포함합니다.

| 이 리뷰가 잡는 것 | Vivado가 잡는 것 |
|-------------------|-----------------|
| 코딩 스타일 위반 | 합성 에러/경고 |
| 포트 폭 불일치 (교차 파일) | 포트 폭 불일치 (단일 파일) |
| 테스트벤치 품질 | — |
| 파라미터화 부족 | — |
| FSM 스타일 | FSM 인식/최적화 |
| XDC ↔ Top 포트 일치 | — |
