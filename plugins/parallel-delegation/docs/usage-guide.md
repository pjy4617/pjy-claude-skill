# Parallel Delegation 플러그인 사용 설명서

## 개요

`parallel-delegation` 플러그인은 Claude Code의 `Agent` 도구를 활용해 **여러 독립 작업을 동시에** 서브에이전트에게 위임하는 워크플로 스킬입니다. 핵심은 단순합니다 — **하나의 응답 메시지 안에 여러 `Agent` 호출을 묶어야 병렬로 실행됩니다**. 호출을 응답 사이로 나누면 순차 실행이 되어 시간 이득이 사라집니다.

이 스킬은 다음 두 가지 함정을 피하기 위한 절차를 제공합니다.

1. **메시지 분리 함정** — 호출을 응답 사이로 나눠 순차 실행이 되는 문제
2. **컨텍스트 누락 함정** — 서브에이전트는 대화 맥락을 모르는 새 인스턴스이므로, self-contained 프롬프트가 아니면 환각·오해·중복 작업이 발생

### 적용 효과

순차로 진행하면 **N개 작업 시간의 합**이 걸리지만, 병렬화하면 **가장 느린 작업 하나의 시간**으로 압축됩니다. 4개 작업이 각 5분이라면 20분 → 5분 (75% 단축).

## 설치

### 사전 요구사항

- Claude Code CLI (서브에이전트 / `Agent` 도구를 지원하는 환경)

### 플러그인 설치

```bash
# pjy-skills 마켓플레이스에서 설치
claude marketplace install parallel-delegation
```

### 3-role 에이전트 설치 (권장)

타겟 프로젝트에서 3-role 에이전트(구현/테스트/구현품질검증)를 사용하려면:

```
/parallel-delegation-setup
```

- 기존 `.claude/agents/`와 OMC 설치 여부를 자동 스캔
- 누락된 역할만 번들에서 보충 (사용자 정의 에이전트는 덮어쓰지 않음)
- OMC 사용자는 OMC 에이전트(`executor`/`test-engineer`/`verifier`)를 그대로 사용 가능

## 스킬 목록

| 스킬 | 명령어 | 기능 |
|------|--------|------|
| 병렬 위임 워크플로 | `/parallel-delegation` | 다중 작업을 단일 메시지에 Agent 호출 묶음으로 동시 실행 |
| 3-role 에이전트 설치 | `/parallel-delegation-setup` | 구현/테스트/구현품질검증 에이전트를 누락분만 보충 설치 |

이 워크플로 스킬은 사용자가 명시적으로 호출하지 않아도 다음 트리거에 자동으로 활성화됩니다.

- `"A 하고 B 하고 C 해줘"` — 다중 의도
- `"프론트엔드와 백엔드를 동시에"`, `"한꺼번에"`, `"병렬로"`
- `"subagent 써서"`, `"team으로"`, `"multi-agent"` 언급
- 서로 다른 파일/도메인을 한 요청에서 다룸
- 탐색 + 구현이 동시에 가능한 상황

## 3-role 분리 패턴 (구현 / 테스트 / 구현품질검증)

같은 에이전트가 구현·테스트·검증을 모두 하면 **자기 검증의 함정**(자기 코드의 약점을 자기가 못 봄)에 빠집니다. parallel-delegation은 다음 3-role 분리를 권장합니다.

| 역할 | 사고방식 | 책임 | 번들 에이전트 | OMC 대응 |
|------|---------|------|------------|---------|
| **구현** | "어떻게 동작하게 할지" | 코드 작성·수정·리팩토링. **테스트·검증 안 함** | `pd-implementer` | `oh-my-claudecode:executor` |
| **테스트** | "무엇이 옳은 동작인지" | 테스트 코드 작성, 엣지 케이스 발굴, TDD. **구현 안 함** | `pd-tester` | `oh-my-claudecode:test-engineer` |
| **구현품질검증** | "사양·통합·품질을 만족하는가" | 사양 정합성, 누락 엣지, 통합 동작, 외부 정합성 검증. **수정 안 함** | `pd-verifier` | `oh-my-claudecode:verifier` |

### 왜 분리하는가

| 같은 에이전트가 다 하면 | 분리하면 |
|--------------------|--------|
| 자기 가정으로 짠 테스트가 자기 가정의 버그를 못 잡음 | 테스트가 구현의 가정을 의심하고 반례를 찾음 |
| "다 됐다"는 자기 보고를 누구도 검증 못 함 | 검증자가 사양·산출물을 독립 시각으로 대조 |
| 단위 통과 = 시스템 동작이라고 착각 | 검증자가 통합·외부 정합성까지 확인 |

### 워크플로 예시

```
사용자: "JWT 인증 미들웨어 만들고 테스트도 짜고 검증해줘"

[메인 에이전트의 단일 응답]
 ├─ Agent(pd-implementer, model="opus", "JWT 미들웨어 구현 — sec-spec.md §3 따를 것")
 ├─ Agent(pd-tester,      model="opus", "JWT 미들웨어 테스트 — 만료/위조/none 알고리즘 케이스")
 └─ Agent(pd-verifier,    model="opus", "구현+테스트 결과를 sec-spec.md와 대조 검증")

[결과 합성]
  메인이 3 보고를 받아 통합. CRITICAL/MAJOR 발견 시 다음 응답에서
  pd-implementer / pd-tester에 재위임.
```

## 에이전트 매칭 메커니즘

### 자동 발견 (Discovery)

Claude Code 세션 시작 시 다음 위치를 스캔해 `subagent_type` 카탈로그를 만듭니다. **사용자가 별도로 등록할 필요 없습니다.**

| 우선순위 | 출처 | 예 |
|---------|------|-----|
| 1 | 프로젝트 `.claude/agents/*.md` | `pd-implementer`, `kicad-schematic-reviewer` |
| 2 | 플러그인 제공 (`<plugin>:<agent>`) | `oh-my-claudecode:executor` |
| 3 | Claude Code 내장 | `Explore`, `general-purpose`, `Plan` |

### 자동 매칭 (Matching)

§2(에이전트 매칭) 단계에서 메인 에이전트가 카탈로그를 훑어 **description 적합도가 가장 높은** 에이전트를 선택합니다. 출처와 무관하게 description만으로 결정됩니다.

> 우선순위는 **이름 충돌 시** 어떤 에이전트가 그 이름을 차지하는지의 해석 순서일 뿐, 매칭 자체는 description 기반입니다.

### description 품질이 매칭 정확도를 결정

사용자 정의 에이전트의 description이 모호하면 매칭이 빗나가 `general-purpose`로 폴백됩니다.

```yaml
# 나쁨 — 매칭 실패 가능
description: "회로도 관련 작업"

# 좋음 — 매칭 정확
description: "KiCad 회로도(.kicad_sch) 8단계 리뷰 — PDF 시각 검토,
  라이브러리 비교, 데이터시트 정합성, ERC. '회로도 리뷰',
  '핀 검증', 'BOM 검토' 요청 시 자동 선택."
```

좋은 description의 조건:
- 역할 한 줄 요약 (무엇을 하는 에이전트인가)
- 트리거 키워드 ("X 요청 시 자동 선택")
- 책임 범위 (포함/제외)

### 사용자 정의 에이전트도 자동 사용된다

`.claude/agents/my-implementer.md`를 만들어두면 자동으로 카탈로그에 등록되고, description이 적합하면 메인이 자동 선택합니다. **번들 에이전트는 빈 역할만 채우는 보조 역할**일 뿐, 사용자 의도를 침범하지 않습니다 — 이를 위해 setup 스킬이 역할 매핑·중복 체크를 수행합니다.

## 8단계 워크플로

### 1단계 — 분해 (Decomposition)

사용자 요청을 **독립 실행 가능한 단위**로 자릅니다.

| 상황 | 결정 |
|------|------|
| A의 결과가 B의 입력 | 순차 (병렬 불가) |
| 서로 다른 파일/도메인 | 병렬 가능 |
| 같은 파일 동시 수정 | 병렬 + worktree 격리 |

### 2단계 — 에이전트 매칭

각 작업에 적합한 `subagent_type`을 고릅니다. 프로젝트의 `.claude/agents/`에 도메인 전문 에이전트가 있다면 우선 사용합니다.

| 작업 유형 | 권장 에이전트 |
|-----------|--------------|
| 코드베이스 탐색 / 심볼 위치 | `Explore` |
| 단순 구현 / 리팩토링 | 일반 구현 에이전트 (executor 등) |
| 복잡한 자율 작업 | 깊이 있는 구현 에이전트 (deep-executor 등) |
| 코드/보안 리뷰 | 리뷰 전문 에이전트 |
| 테스트 작성 / 전략 | 테스트 엔지니어 |
| 빌드 실패 / 타입 오류 | 빌드 픽서 |
| 검증 / 완료 증거 수집 | 검증 에이전트 |

### 3단계 — 모델 라우팅

**기본값은 `opus`이며, 단순 탐색만 `sonnet`을 씁니다.** 비용 최적화보다 품질 일관성을 우선합니다.

| 작업 유형 | 모델 |
|-----------|------|
| 단순 탐색 — 파일 위치, 심볼 찾기, grep | `sonnet` |
| 구현·분석·리뷰·테스트·검증·디버깅·리팩토링 | `opus` |

판단 기준: **읽고 보고만** 하면 sonnet, **판단·작성·수정**을 하면 opus. 의심스러우면 opus.

### 4단계 — 단일 메시지 묶기 ★

병렬화의 결정적 순간입니다. 같은 응답 안에 여러 `Agent` 호출을 동시에 발사합니다.

```
[하나의 응답]
 ├─ Agent(executor,      "UI 컴포넌트 작성")     ┐
 ├─ Agent(executor,      "API 핸들러 작성")       ├─ 동시 시작
 ├─ Agent(architect,     "DB 스키마 설계")        │
 └─ Agent(test-engineer, "통합 테스트 작성")     ┘
```

이 단계를 건너뛰고 호출을 한 개씩 응답 사이로 분리하면, 병렬화 효과가 사라집니다.

### 5단계 — Self-contained 프롬프트 작성

서브에이전트는 **대화 맥락을 모르는 새 인스턴스**입니다. 매 호출마다 다음을 적습니다.

```
[목표] 무엇을 / 왜
[컨텍스트] 관련 파일·함수·이전 시도 (예: src/foo.ts:120)
[제약] 코딩 규칙·표준·금지사항
[산출물] 파일 변경 / 보고서 (분량·형식)
[모드] 연구만 vs 코드 작성
```

#### 좋은 예

```
[목표] src/auth/ 아래에 JWT 검증 미들웨어 추가
[컨텍스트] 기존 src/middleware/logger.ts:42의 미들웨어 등록 패턴 따를 것.
           라우터 부착 위치는 src/server.ts:120
[제약] 프로젝트 인증 사양 §X.Y 준수, 외부 라이브러리 금지
[산출물] src/auth/jwt_middleware.ts 신규 파일 작성
[모드] 코드 작성. 테스트는 별도 에이전트 담당.
```

#### 나쁜 예

```
"로그인 만들어줘"
```

→ 서브에이전트가 어떤 인증 방식, 어디에, 어떤 규칙으로 만들어야 할지 알 수 없어 환각합니다.

### 6단계 — Worktree 격리 결정

같은 파일 영역을 여러 에이전트가 수정해야 한다면 `isolation: "worktree"`를 추가합니다. 각 에이전트가 독립 Git worktree에서 작업해 충돌을 막습니다. 변경이 없으면 worktree는 자동 정리됩니다.

| 상황 | 격리 |
|------|------|
| 서로 다른 파일/디렉토리 작업 | 격리 불필요 (오버헤드 ↓) |
| 같은 파일/모듈 동시 수정 | `isolation: "worktree"` 필수 |

### 7단계 — Foreground vs Background

| 모드 | 사용 상황 |
|------|-----------|
| **Foreground (기본)** | 결과를 받아야 다음 단계 진행 — 리서치, 설계 검토, 동기적 산출물 |
| **Background** (`run_in_background: true`) | 메인이 다른 일 하는 동안 돌리기 — 빌드, 긴 테스트, 독립 사이드 작업 |

Background로 돌렸다면 **폴링하지 않습니다**. 완료 알림을 기다립니다 (`sleep` / `while` 폴링 금지).

### 8단계 — 합성과 검증

서브에이전트가 반환하는 것은 **요약**입니다. 메인의 책임은 다음과 같습니다.

1. **검증** — 요약을 그대로 믿지 말고 실제 파일 변경을 직접 확인 (`git diff` 등)
2. **충돌 해소** — 여러 에이전트 결과 간 중복·모순 정리
3. **통합 보고** — 사용자에게는 한 덩어리로 — raw 서브에이전트 출력을 그대로 노출하지 않음

## 실전 예시

### 예시 1 — 풀스택 기능 구현

요청: *"사용자 프로필 페이지 추가 — 프론트엔드 컴포넌트, API 엔드포인트, 마이그레이션, e2e 테스트까지"*

```
[하나의 응답]
 ├─ Agent(Explore,       model="sonnet", "기존 프로필 관련 코드/테이블 위치 파악")
 ├─ Agent(executor,      model="opus",   "src/components/UserProfile.tsx 작성")
 ├─ Agent(executor,      model="opus",   "src/api/profile.ts API 핸들러 추가")
 ├─ Agent(architect,     model="opus",   "DB 마이그레이션 설계 + SQL 작성")
 └─ Agent(test-engineer, model="opus",   "e2e 시나리오 + Playwright 테스트")
```

5개 작업을 단일 응답으로 발사 → 가장 느린 하나의 시간만큼만 소요.

### 예시 2 — 멀티 도메인 리팩토링

요청: *"인증 모듈을 JWT에서 OAuth로 리팩토링하고, 영향받는 테스트도 모두 업데이트"*

```
[하나의 응답]
 ├─ Agent(executor,      model="opus", "src/auth/* OAuth로 전환",     isolation: "worktree")
 └─ Agent(executor,      model="opus", "tests/auth/* 업데이트",        isolation: "worktree")
```

같은 영역을 동시에 만지므로 worktree 격리. 메인이 결과를 받아 머지·검증.

### 예시 3 — 리서치 + 구현 동시 진행

요청: *"이 라이브러리 사용법 조사하고, 기본 통합 코드까지 작성"*

```
[하나의 응답]
 ├─ Agent(document-specialist, model="opus",   "라이브러리 공식 문서·예제 수집 → 보고서")
 └─ Agent(Explore,             model="sonnet", "코드베이스에서 통합 위치 후보 탐색")

[그 다음 응답 — 위 결과를 받아서]
 └─ Agent(executor, model="opus", "통합 코드 작성", [위 두 보고서 인용])
```

리서치는 병렬, 구현은 결과 의존이므로 순차.

## 흔한 함정과 대응

| 함정 | 증상 | 대응 |
|------|------|------|
| 호출을 응답 사이로 분리 | 병렬이 아니라 순차 실행 | 한 응답에 묶기 |
| 프롬프트에 "위에서 본 것처럼" | 서브에이전트가 맥락 없어 헤맴 | 매 호출마다 컨텍스트 재기술 |
| 탐색 작업까지 opus로 호출 | 불필요한 비용·지연 | `Explore` 호출만 `sonnet`으로 분리 |
| 서브에이전트 요약을 그대로 사용자에게 | 거짓 완료 보고 위험 | 실제 파일 변경을 직접 검증 후 통합 |
| 병렬화에 집착 | 의존성 있는 작업까지 병렬 시도 | 의존성 있으면 순차로 짜기 |
| 같은 파일 동시 수정 | 파일 충돌·덮어쓰기 | `isolation: "worktree"` 적용 |

## 사용 시기 결정

### 적용해야 할 때

- 사용자가 다중 의도를 표명 (`"A 하고 B 하고 C"`)
- 프론트/백/DB/테스트 등 서로 다른 도메인 동시 작업
- 탐색(Explore)과 구현이 동시에 가능
- 리뷰 + 구현 + 테스트 작성이 독립적으로 진행 가능
- 사용자가 "병렬", "동시", "한꺼번에", "team", "subagent" 명시
- 단일 에이전트로 진행하면 한참 걸릴 작업

### 적용하지 말아야 할 때

- 작업이 본질적으로 순차 (A → B → C 의존)
- 단순 한 줄 수정, 한 파일 읽기
- 사용자가 직접 단계별 확인을 원함

---

## 실패 복구

- **한 에이전트만 실패** — 다른 결과는 살리고, 실패한 작업만 재시도
- **재시도 시** — 같은 프롬프트를 그대로 반복하지 말고 부족했던 컨텍스트(파일 경로, 제약, 예시)를 추가
- **진행 중인 에이전트와 이어서 작업** — 새 `Agent` 호출이 아니라 `SendMessage({to: name})`로 컨텍스트를 이어받음

## 다른 플러그인과의 관계

이 스킬은 **워크플로 스킬**입니다 — 도메인 작업(회로도, 펌웨어 등)을 직접 수행하지 않고, 다른 도메인 작업을 **어떻게 병렬로 위임할지**의 절차만 제공합니다. 다음 도메인 플러그인과 함께 쓸 때 효과가 큽니다.

| 플러그인 | 병렬화 시나리오 예시 |
|----------|--------------------|
| `vivado` | RTL 설계 + 시뮬레이션 + XDC 검증을 동시 위임 |
| `stm32f4` | 펌웨어 구현 + 회로도 검증 + 테스트 작성을 동시 위임 |
| `wmx3md` | 3-layer 모듈(Core / API / gtest)을 각 레이어별 에이전트에게 동시 위임 |
| `kicad` | 회로도 설계 + 데이터시트 수집 + BOM 생성을 동시 위임 |
| `manual` | 한국어 / 영어 / 일본어 매뉴얼을 동시 생성 |

## 자주 묻는 질문

### Q: 병렬 호출 개수에 제한이 있나요?

기술적 상한은 환경에 따라 다르지만, 실용적으로는 **3~6개**가 가장 안정적입니다. 너무 많이 묶으면 결과 합성이 복잡해지고, 토큰 비용도 늘어납니다.

### Q: 모든 호출이 같은 모델이어야 하나요?

아닙니다. 호출별로 `model`을 따로 지정할 수 있습니다. 탐색은 `sonnet`, 구현·분석은 `opus`로 섞어 쓰는 것이 권장 패턴입니다.

### Q: 백그라운드 에이전트가 끝났는지 어떻게 알 수 있나요?

자동으로 완료 알림이 옵니다. **수동으로 폴링하거나 sleep으로 기다리지 마세요** — 알림을 기다리며 다른 작업을 하면 됩니다.

### Q: worktree 격리는 항상 켜는 게 좋지 않나요?

오버헤드가 있습니다 (worktree 생성/정리 시간, 디스크 I/O). 같은 파일을 동시 수정하지 않는다면 격리 없이 본 워크트리에서 작업하는 게 더 빠릅니다.

### Q: 서브에이전트의 보고서를 그대로 사용자에게 전달해도 되지 않나요?

권장하지 않습니다. 서브에이전트의 요약은 **의도한 것**을 묘사할 뿐, 실제로 일어난 것을 보장하지 않습니다. 메인 에이전트가 `git diff` 등으로 직접 검증한 뒤 통합된 결론으로 전달해야 합니다 (trust but verify).
